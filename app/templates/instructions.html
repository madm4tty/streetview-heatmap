Project Overview

What is Street View Heatmap?

Street View Heatmap is a tool that visualizes the age of Google Street View imagery across the UK. It helps identify areas where Street View coverage is outdated or missing, which is valuable for:

  • Urban planners - understanding infrastructure documentation status
  • Researchers - analyzing temporal coverage patterns
  • Mapping enthusiasts - discovering where new imagery might be captured soon
  • Local authorities - identifying areas needing updated visual documentation

Why Does Image Age Matter?

Google Street View imagery varies significantly in age. Some areas have imagery updated every few months, while others may not have been updated for years. Older imagery may show:

  • Buildings that have been demolished or significantly changed
  • Road layouts that no longer exist
  • Businesses that have closed or moved
  • Areas before major development projects

How Data is Collected

The system works by:

  1. Querying OpenStreetMap - Using the Overpass API to get road network data
  2. Sampling locations - Selecting points along each road segment
  3. Checking Street View - Using Google's Street View Metadata API to find capture dates
  4. Storing results - Saving location data with capture dates in PostgreSQL

Understanding the Map

Color Scheme

Roads are colored based on how recently Google captured Street View imagery at that location:

Green - Fresh

Less than 3 months old

Yellow - Recent

Less than 1 year old

Orange - Dated

Less than 3 years old

Red - Stale

3 or more years old

Coverage Grid

Toggle the "Coverage Grid" layer to see the tile boundaries:

  • Blue outline - Tiles with collected data
  • Gray dashed outline - Tiles without data yet

Interacting with the Map

  • Pan - Click and drag to move around
  • Zoom - Use mouse wheel or +/- buttons
  • Search - Type a city name to jump to that location
  • Hover - Move over roads to see quick info
  • Click - Click on roads for detailed information
Note: Road data only loads when zoomed in to level 10 or higher. Zoom in to see individual road segments.

API Information

Google Street View Metadata API

This project uses Google's Street View Metadata API, which is free and unlimited. Unlike the Street View Image API (which costs money per image), the Metadata API only returns information about availability and capture dates without actually serving imagery.

Cost: Free - no charges for metadata requests

Overpass API (OpenStreetMap)

Road network data comes from OpenStreetMap via the Overpass API:

  • Free to use with reasonable rate limiting
  • Comprehensive road network data for the UK
  • Includes road names, types, and geometries

Rate Limiting

To be respectful of external APIs, the system includes built-in delays:

  • Overpass API - 2 second delay between requests (configurable)
  • Street View API - Concurrent requests limited to 20 (configurable)

Data Freshness

Update Schedule

The system can run automatic updates on a configurable schedule (default: every 24 hours). Updates are "smart" and prioritize:

  1. High priority areas - Major cities and urban centers
  2. Medium priority areas - Towns and major roads
  3. Low priority areas - Rural areas and minor roads

Smart Refresh Strategy

To avoid redundant API calls, the system:

  • Skips locations checked within the last 90 days (configurable)
  • Prioritizes areas that haven't been checked recently
  • Batches requests efficiently to maximize coverage

Manual Updates

From the Dashboard, you can trigger manual updates with specific priority filters:

  • Update only high-priority areas for quick city coverage
  • Update all priorities for comprehensive refresh
  • Set tile limits to control processing time

Geographic Coverage

UK Mainland Coverage

The system covers the UK mainland with approximately 44,000 tiles, each roughly 5km x 5km (0.05° x 0.05°).

Priority Cities

High-priority areas include major UK cities:

London Birmingham Manchester Glasgow Liverpool Leeds Edinburgh Bristol Sheffield Newcastle Cardiff Belfast

Road Type Priorities

Within each area, roads are prioritized by type:

  • Primary roads - A-roads, major routes
  • Secondary roads - B-roads, important local roads
  • Residential roads - Local streets in populated areas

Why Some Roads Have No Data

Some roads may appear without coverage data because:

  • Google hasn't captured Street View there yet
  • The area is private or restricted access
  • The tile hasn't been processed yet
  • The road is too new for OpenStreetMap data

Technical Details

Technology Stack

Backend

  • Flask - Python web framework
  • PostgreSQL - Database with PostGIS
  • APScheduler - Background job scheduling
  • aiohttp - Async HTTP requests

Frontend

  • Leaflet - Interactive maps
  • Vanilla JavaScript - No framework dependencies
  • CSS Variables - Theming and responsiveness

Data Storage

  • Tile size: 0.05° x 0.05° (~5km x 5km)
  • Storage format: PostgreSQL with GeoJSON support
  • Capacity: Millions of location records

Server Configuration

Port: The application runs on port 5001 (not 5000) to avoid conflicts with macOS AirPlay Receiver which uses port 5000.

API Documentation

The application provides a REST API for programmatic access.

Base URL

http://localhost:5001/api

Endpoints

GET /api/health

Check API health status

curl http://localhost:5001/api/health

GET /api/status

Get comprehensive system status including coverage statistics

curl http://localhost:5001/api/status

GET /api/tiles

List tiles with optional filtering

# Get all tiles
curl http://localhost:5001/api/tiles

# Filter by priority
curl "http://localhost:5001/api/tiles?priority=high"

# Filter by bounding box
curl "http://localhost:5001/api/tiles?min_lat=51.4&max_lat=51.6&min_lon=-0.2&max_lon=0.1"

GET /api/tiles/{tile_id}/data

Get GeoJSON data for a specific tile

curl http://localhost:5001/api/tiles/51.50_-0.15/data

POST /api/update/trigger

Trigger a manual update (requires API key)

curl -X POST http://localhost:5001/api/update/trigger \
  -H "Content-Type: application/json" \
  -H "X-API-Key: your-api-key" \
  -d '{"priority_filter": "high", "tile_limit": 10}'

GET /api/config

Get current configuration

curl http://localhost:5001/api/config

POST /api/config

Update configuration (requires API key)

curl -X POST http://localhost:5001/api/config \
  -H "Content-Type: application/json" \
  -H "X-API-Key: your-api-key" \
  -d '{"scheduler": {"enabled": true}, "update": {"batch_size": 100}}'

Authentication

Endpoints that modify data require an API key passed in the X-API-Key header. The API key is configured in the application's config.yaml file.

Response Format

All responses are JSON. Error responses include an error field with a description.
